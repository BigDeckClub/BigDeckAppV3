Optional frontend change (Option B) — replace the GET redirect with a POST fetch (only if you prefer the client to send POST):

```javascript name=src/components/LoginPage.jsx
// Replace the current onClick that sets window.location.href = '/api/login'
// with a POST-based flow. Integrate into your existing component.

function handleSignIn() {
  // Example: send credentials in body (adjust to your auth flow)
  fetch('/api/login', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ username: '...', password: '...' }) // replace with real inputs
  })
    .then(async (res) => {
      if (!res.ok) {
        const err = await res.json().catch(() => ({ message: 'Login failed' }));
        throw new Error(err.message || 'Login failed');
      }
      return res.json();
    })
    .then((data) => {
      // handle successful login (redirect, update UI, store token, etc.)
      console.log('Logged in:', data);
      window.location.href = '/'; // or other post-login route
    })
    .catch((err) => {
      console.error('Login error:', err);
      // show error message in UI
    });
}

// Usage in JSX (replace the existing button handler):
// <button onClick={handleSignIn}>Sign In Now</button>
```

  2. Visit /api/login in the browser (or run curl http://localhost:PORT/api/login). You should get JSON: { "message": "Use POST with credentials" } instead of the "endpoint not found" 404.
  3. Click the Sign In button on the login page — the page should no longer show "API endpoint not found". If your UI expects a redirect or token, you may need to update additional logic to match how the endpoint responds.

- If you apply Option B (frontend POST):

Notes and recommendations
- The minimal GET handler is safe and prevents the 404, but it may only be a stopgap. If your intended flow is OAuth (redirect to Replit), you should change the GET handler to redirect to the OAuth provider (res.redirect(REPLIT_OAUTH_URL)).
- If the POST handler implements credential login, Option B (switching frontend to POST) makes the flow semantically correct and is more RESTful. Use whichever better matches your auth design.
